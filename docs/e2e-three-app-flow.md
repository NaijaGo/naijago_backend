# Three-App End-to-End Manual Test

Use this checklist after backend, admin, customer, vendor, and rider changes. The backend is the source of truth for orders, OTPs, rider assignment, app notifications, push notifications, and admin logs.

## Prerequisites

- Backend API is running against the test database.
- Admin panel points to the same backend `BASE_URL`.
- Customer app, vendor app, and rider app are installed or running against the same backend.
- OneSignal app IDs and keys are configured for customer, vendor, and rider apps.
- Mapbox token is configured in the rider app where rider maps are used.
- A test customer, approved vendor, approved rider, and test product are available.

## Flow

1. Customer places an order from the customer app.
2. Confirm the vendor app receives the new order notification.
3. Vendor accepts/prepares the order, then marks it ready.
4. Confirm the backend creates or exposes the rider assignment queue.
5. Rider accepts the delivery in the rider app.
6. Confirm customer, vendor, rider, and admin order views show the rider assignment.
7. Rider arrives for pickup and enters or scans the pickup OTP generated by the backend.
8. Confirm pickup status updates in the backend, vendor app, rider app, customer app, and admin panel.
9. Rider completes delivery using the delivery OTP generated by the backend.
10. Confirm final delivery status, payout/wallet impact, and order completion in all apps/admin.

## Notification Checks

- Customer receives order status notifications inside the customer app.
- Vendor receives order, ready/pickup, and completion notifications inside the vendor app.
- Rider receives assignment and OTP/status notifications inside the rider app.
- Admin `Notifications` page shows delivery/reporting stats.
- Admin `Notification History` includes in-app, push, email, and WhatsApp campaign logs where applicable.

## OTP Checks

- Pickup OTP and delivery OTP come from backend order data only.
- OTPs appear in the correct app notification/context for the correct role.
- Wrong OTP is rejected.
- Correct OTP can be used once and moves the order to the next state.

## Failure Checks

- Restart the API while a notification is scheduled; the worker should still send it.
- Put a scheduled notification into a stale `sending` state in test data; the worker should recover it to `scheduled` and process it.
- Turn off OneSignal temporarily; app notification records should still be stored, while push failures appear in logs.
